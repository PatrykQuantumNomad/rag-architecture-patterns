## Tech Debt (Non-Blocking)

The following items are documented in phase summaries and tracked for v1.1. None block milestone closure.

### LiteLLM judge cost token-parser gap (Phase 7, Phase 8)
All 5 `ragas-judge-tier-{N}-*.json` cost ledgers report `totals.usd=0.0` because LiteLLM does not parse OpenRouter token-usage responses. **Real spend** is ~$0.45 across all 5 tiers per sweep; honest Phase 7 total is `$0.439` (well under `$3.00` ceiling). **Mitigated in Phase 8** via `_estimate_cost_fallback` which writes an `estimator: {estimated:true, method:fixed_per_cell, ...}` block when in_tok==0. Phase 7 SC-5 was human-verified by dashboard check on 2026-05-07.

### `tracker.persist()` ignores `--results-dir` (Phase 5)
`run.py:273` and `score.py:518` call `tracker.persist()` without forwarding the caller's `--results-dir` / `--output-dir`, so cost files always land in the hardcoded `DEFAULT_COSTS_DIR`. Tests work around via monkeypatch. Documented in 05-02-SUMMARY.md as a v1.1 hardening item.

### RAGAS 0.4.3 deprecation warnings (Phase 3)
`from ragas.metrics import context_precision` triggers `DeprecationWarning` (RAGAS API evolution). Pre-existing; tracked as v1.1 migration.

### `pyproject.toml addopts` not deselecting live tests by default (Phase 3)
Bare `pytest evaluation/tests/test_eval_smoke_live.py -v` can accidentally invoke API-budget-consuming tests. Tracked as v1.1 hardening.

### Pytest test-bleed cost JSON pollution (Phase 7)
5 zero-byte `tier-{N}-eval-20260507T120542Z.json` placeholder cost JSONs from a post-sweep pytest run live in `evaluation/results/costs/`. They get picked up by `ls -t | head -1` mtime-freshest scans, causing $0 totals to appear in automated cost rollups. The honest-total path uses timestamp-window selection. Documented in 07-03-SUMMARY.md.

### `comparison.md` Embedder section uses bold instead of heading (Phase 7)
`compare.py` emits `**Embedder by tier:**` (markdown bold) rather than `## Embedder by tier` (heading). No user-visible impact; the table content is correct. The plan's regex spec was wrong; compare.py was RAW-LOCKED so no fix applied.

### Stale REQUIREMENTS.md checkboxes (admin)
REQUIREMENTS.md still shows `TIER-03 [~]`, `CAP-01 [ ]`, `CAP-03 [ ]` even though all three are SATISFIED per VERIFICATION.md. Phase 7 verifier explicitly notes "REQUIREMENTS.md still shows CAP-01 as `[ ] Pending` — this is expected as the checklist update is a post-verification orchestrator action, not a plan deliverable." Will be resolved at milestone archive (REQUIREMENTS.md is deleted and replaced by `milestones/v1.0-REQUIREMENTS.md` with all items marked complete).
