## Validation Architecture

Phase 7 is non-code-bearing. There is no RED→GREEN cycle. Validation is
two-layered:

1. **Per-task `<verify><automated>` blocks** — every task in plans 07-01 /
   07-02 / 07-03 ends with an `<automated>` shell assertion that the task's
   stated postcondition holds. These are the per-step gates.
2. **Phase-level inline-bash verifier (`07-03-VERIFY.txt`)** — the
   load-bearing artifact. Generated by 07-03 Task 4 by tee'ing the verifier
   block into a transcript file. The `OVERALL VERDICT: PASS` line at the
   bottom is the single load-bearing assertion that Phase 7 met all 5
   ROADMAP success criteria. **Absence of the line, OR presence of any
   `SC-N: FAIL` line, is the verdict regardless of the bash pipeline's exit
   code** (per 07-03 Task 4 design — `tee` masks the inner exit, so the
   grep over the transcript is what gates the task). 07-03 Task 6's
   regression check then asserts the 6 forward-contract-locked harness
   modules + 3 tier-4 helper scripts remain byte-identical.

Verbatim transcript preservation: `07-03-VERIFY.txt` is committed alongside
`07-03-SUMMARY.md` in 07-03 Task 6's commit so the verdict's evidence
survives in git history.

## Cross-Plan Data Contract — Tier 4 Manifest Path

07-03 Task 2 needs to read the **full-rebuild** Tier 4 graph-stats manifest
(NOT the 3-paper smoke manifest at
`evaluation/results/diagnostics/tier-4-graph-stats-2026-05-05T11_14_40Z.json`)
to confirm the captured graph is the full-corpus rebuild.

**Risk avoided:** A naive `ls -t evaluation/results/diagnostics/tier-4-graph-stats-*.json | head -1`
picks the freshest by mtime. If the smoke manifest's mtime is somehow newer
than the full-rebuild manifest's (e.g. the smoke manifest was touched after
07-02 ran), the wrong file is read and the `NODES > 10000` guard would fail
mistakenly OR — worse — the smoke manifest would pass it incorrectly if its
node count happened to read as something >10000.

**Contract (cross-plan, single source of truth):**

- **07-02 Task 5 emits the new manifest path in `07-02-SUMMARY.md`
  front-matter** (key: `tier_4_full_rebuild_manifest`). Example:
  ```yaml
  tier_4_full_rebuild_manifest: evaluation/results/diagnostics/tier-4-graph-stats-2026-05-06T18_22_47Z.json
  ```
- **07-03 Task 2 reads that key** before doing the full-corpus sanity check.
  If 07-02-SUMMARY.md is missing the key, 07-03 Task 2 falls back to the
  explicit-filename guard (`[[ "$NEWEST_MANIFEST" != *'2026-05-05T11_14_40Z'* ]] || fail`).

This eliminates the mtime-ambiguity failure mode and makes the cross-plan
dependency explicit.

## Notes for Reviewers

**Why no pytest tests?** Phase 7 ships zero source code. The 6
forward-contract-locked harness modules + 3 tier-4 helper scripts are
RAW-LOCKED for the entire phase (verified by `git diff HEAD -- ... | wc -c == 0`
at the end of each plan). New pytest tests would imply new behavior; new
behavior would imply source edits; source edits would break the
forward-contract guard. So the validation surface is necessarily runtime
shell + jq, not pytest.

**Why is the verifier inline bash, not a phase-local Python helper?**
07-RESEARCH.md §"Claude's Discretion" prefers inline for one-shot
verification; a Python helper makes sense only when the verifier is reused
across plans/phases. Phase 7's verifier runs ONCE per sweep. Phase 8
(multi-judge spot-check) and Phase 9 (frozen handoff doc) will write
their own verifiers.

**Why does the verifier's exit code not gate the task?** 07-03 Task 4's
verifier runs inside a `{ ... } 2>&1 | tee 07-03-VERIFY.txt` pipeline. The
outer pipeline's exit code is `tee`'s exit code (almost always 0). The
inner `exit 1` short-circuits the block but does not propagate through tee.
The load-bearing assertion is therefore the post-tee
`grep -q "OVERALL VERDICT: PASS" 07-03-VERIFY.txt`. This is by design and
documented in 07-03 Task 4's `<action>` and `<verify>` prose so executors
do not "fix" the tee pattern (which would lose the verbatim transcript).
