<objective>
Run the live full sweep that closes CAP-01: 5 tiers × 30 questions captured
and scored on a single date with a single git SHA recorded in every per-tier
QueryLog. Verify the result against the 5 ROADMAP success criteria. ZERO
source code changes; ZERO harness file edits.

Purpose: This is the operational core of Phase 7. Plan 07-01 cleared
pre-flight; Plan 07-02 rebuilt the Tier 4 graph; Plan 07-03 produces the
numbers that feed Phase 8 (multi-judge spot-check) and Phase 9 (frozen
handoff doc). The sweep itself is one pipeline.py command — the rest of the
plan is sequencing discipline + post-sweep verification + a human-verify
checkpoint that confirms numbers look plausible before triggering
/gsd-verify-phase.

Three load-bearing pieces beyond "just run pipeline.py":
1. **Step 1 first**: Tier 4 must be captured via eval_capture.py BEFORE
   pipeline.py runs (run.py:233-247 returns yellow-skip if --tier-4-from-cache
   is unset; Phase 130 SC-1 deferred this).
2. **No commits between Step 1 and Step 2**: per Pitfall 2 of 07-RESEARCH.md
   — pipeline.py reads HEAD via `run._git_sha()` at amain entry; if a commit
   slips in between, the records and the QueryLog SHA disagree at the
   semantic level (passthrough rewrite preserves the QueryLog-level
   invariant but the records were generated against an older code state).
3. **SHA invariant scoped to 5 queries JSONs**: per 07-RESEARCH.md
   §"Single-date invariant", only the 5 QueryLog files carry `git_sha`.
   Cost JSONs carry `tier`+`timestamp`+`totals` only; metrics JSONs carry
   per-row data with no top-level provenance. The ROADMAP Success Criterion
   2 wording "grep all 10 output JSONs" is a slight overstatement —
   grep across the 5 queries JSONs is the load-bearing claim. Plan 07-03's
   verifier honors this honestly.

Output:
- 5 fresh per-tier queries+metrics+costs JSONs all carrying sweep_sha (gitignored)
- Regenerated `comparison.md` with 5-tier rollup + per-class table +
  embedder-by-tier table (Phase 6 D-Q1)
- Post-sweep verifier verdict = PASS recorded in 07-03-SUMMARY.md
- Phase 7 closure handoff to Phase 8

CRITICAL CONSTRAINT — RAW-LOCK: Forward-contract-locked modules
(pipeline.py / run.py / score.py / compare.py / freeze.py / smoke_gate.py)
remain byte-identical. Phase 7 ships zero source code changes.

CRITICAL CONSTRAINT — DO NOT FREEZE: Phase 7 does NOT call `pipeline.py
--freeze`. Phase 9 owns freeze after Phase 8's multi-judge data lands.
This plan deliberately omits any `--freeze` flag from the sweep command.

NOTE — TASK COUNT (6 tasks, exceeds the 5-task generic guideline): Plan
07-03 carries 6 tasks (1 cost-ack + 4 auto + 1 human-verify). The
generic plan-task ceiling is 5; the 6th task here is justified by the
operational, non-code-bearing nature of this phase. All 4 `auto` tasks
are CLI invocations + file I/O assertions (no test design, no
implementation, no codebase exploration). Their cumulative context cost
is modest because each task's `<action>` is a deterministic shell
script with explicit `<verify><automated>` postconditions. Splitting
into two plans was rejected because the sequencing is rigid (Step 1 →
Step 2 → verifier → human-verify → close-out) and a plan boundary
would force a SUMMARY round-trip in the middle of a single sweep
operation, which is precisely what 07-RESEARCH.md Pitfall 2 warns
against (commits between Tier 4 capture and pipeline.py invocation).
The 6-task structure is therefore the correct decomposition for the
operational nature of this phase; this note pre-empts the warning
re-surfacing in any subsequent checker pass.
