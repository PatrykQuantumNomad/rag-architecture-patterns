# Feature Research

**Domain:** Frozen handoff "numbers doc" for a 5-tier RAG comparison blog post (markdown artifact, copy-pasteable across repos)
**Researched:** 2026-05-04
**Confidence:** MEDIUM-HIGH (academic precedent for Model Cards / BenchmarkCards is well-established and HIGH-confidence; mapping to a blog-handoff context required synthesis from RAG-eval blog conventions, which is MEDIUM-confidence)

## Scope Note

This file does **not** catalogue generic RAG features. It catalogues what makes the *frozen markdown numbers doc itself* (`evaluation/results/eval-numbers-v1.X.md`) credible and useful when copy-pasted into the external blog post. The "users" are technical blog readers evaluating RAG architecture choices — they will dismiss numbers without provenance and trust numbers with full disclosure.

**Closest formal precedents found:**
- **Model Cards** (Mitchell et al. 2018, arXiv 1810.03993) — sections: Intended Use, Metrics, Evaluation Data, Quantitative Analyses, Caveats & Recommendations
- **BenchmarkCards** (arXiv 2410.12974, 2024) — explicit format for documenting LLM benchmarks; fields include Intended Use, Dataset, Metrics, Limitations, Baseline Performance, and crucially **Self-Grading Bias disclosure** for benchmarks where the judge model overlaps the model under test
- **RAGAS paper** (arXiv 2309.15217) — published its results as a small markdown-style table with Faithfulness / Answer Relevance / Context Relevance columns and named the judge LLM (`gpt-3.5-turbo-16k`) and embedder (`text-embedding-ada-002`) explicitly
- **LangChain Public Benchmarks** — table columns: Cognitive Architecture | Test (model+version+date) | Score | Cosine Dist | Run Link; clickthrough to LangSmith run for full reproducibility
- **Anyscale LLMPerf** — reproducibility achieved by open-sourcing the harness rather than embedding all provenance in tables; reports percentiles (P50/P90/P95/P99), not just means, and pins tokenizer + input distribution

The good news: our existing `comparison.md` (auto-generated by `evaluation/harness/compare`) already hits most table-stakes — judge model, judge embedder, per-tier capture timestamp, per-tier git SHA, per-tier generation model, NaN breakdown, and four named honest disclaimers. The frozen v1.X doc largely needs to *lock down*, *re-organize for blog readability*, and *add a few items the auto-generated comparison.md does not yet include* (versioned filename, per-tier hardware/sandbox notes, multi-judge spot-check, per-tier links to raw artifacts).

## Feature Landscape

### Table Stakes (Readers Expect These — Missing = Numbers Get Dismissed)

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **Versioned, immutable filename** (`eval-numbers-v1.23.md`, semver-like or eval-date-stamped) | "Frozen" means the file the blog cites cannot be silently rewritten. A reader hitting a broken or different number on re-visit destroys trust. Hugging Face diffusion-bench and BeIR both pin a version string. | LOW | Existing `evaluation/results/comparison.md` is *regenerated*, not frozen — the v1.X file is the snapshot copy. The version number should live in both the filename and an H1/H2 header inside the doc. |
| **Single eval-date for all 5 tiers** | Comparing tier-1 (run 2026-05-02) vs tier-5 (run 2026-04-12) is apples-to-oranges if upstream models drifted (Gemini Flash patches silently). Blog readers will spot this in provenance and dismiss. The v1.0 milestone explicitly calls this out as scope. | LOW (gating constraint, not a doc feature — but the doc must surface it) | Add an "Evaluation date: 2026-05-XX (all 5 tiers, single session)" line in the header so it's the first thing a skeptical reader sees. |
| **Per-tier provenance row** (timestamp, git SHA, generation model + version, judge model + version, judge embedder) | RAGAS paper, LangChain benchmarks, and Model Cards all require this. The §"Reporting Standards" finding from the ReproRAG framework (arXiv) is explicit: "When only the architecture, such as GPT-4 or GPT-3.5, is reported but not the version, we will not be able to reproduce the results." | LOW | Already in `comparison.md` — port verbatim. **Add model version pin** where available (e.g., `gemini-2.5-flash-001` not just `gemini-2.5-flash`); OpenRouter passthrough may obscure this — check and document if version is unknown. |
| **Judge LLM explicitly named in header, not buried** | LangChain benchmarks bury this; readers complain. RAGAS paper puts it in §"Experimental Setup". For a blog handoff, it must be in the first 10 lines so a skim catches it. | LOW | Existing `comparison.md` already does this (`**Judge LLM:** google/gemini-2.5-flash` line 37). Keep the prominence. |
| **Self-grading / family-bias disclaimer** | Multiple 2024-25 papers establish that family-bias is a *separate* phenomenon from self-bias and is not mitigated by "different model, same family" (arXiv 2508.06709, arXiv 2410.21819). When judge=Gemini-Flash and tiers 2-4 generate with Gemini-Flash, this MUST be disclosed or the credibility floor collapses. | LOW (writing) / MEDIUM (mitigation deferred per PROJECT.md) | Existing disclaimer line 58 ("The judge LLM was the same Gemini Flash family Tiers 2-4 use for generation — self-grading bias is acknowledged.") is correct framing. **Strengthen** by linking to one of the cited bias papers and noting which tiers are affected (2, 3, 4) and which are not (1 uses OpenRouter-routed embeddings, 5 uses different judge endpoint per `comparison.md` line 46). |
| **Sample-size disclaimer with explicit wording** | n=30 per tier is below the threshold for statistical-significance testing on RAGAS metrics (which exhibit ~5-15% LLM-judge variance run-to-run per RAGAS reproducibility issues). Honest reporting frames numbers as **magnitudes, not p-values**. | LOW | Existing line 56 is good ("30 questions × ≤5 tiers is too small for statistical-significance testing... magnitude (not p-values) is what the blog post discusses"). Keep verbatim. |
| **NaN / empty-context breakdown per tier** | When a tier reports "—" instead of a number, readers must know whether it's "0% faithfulness" or "the harness couldn't score this tier." Current `comparison.md` does this (lines 49-50). Critical for not letting tier-4/5 look unfairly bad. | LOW | Already done. Document the *cause* (empty_contexts) and link to the post-mortem if one exists in `.planning/`. |
| **Per-question-class breakdown** (single-hop / multi-hop / multimodal × tier) | A single rolled-up mean hides which question types each tier wins. The blog narrative depends on showing tier-3 dominates structured retrieval and tier-4 (when fixed) wins multimodal. Without the breakdown, tier rankings look one-dimensional. | LOW | Already generated in `comparison.md` lines 17-33. Port to v1.X. |
| **Cost + latency alongside quality** | RAG architecture choice is a 3-axis tradeoff (quality / cost / latency). A faithfulness-only table is misleading. Anyscale's LLMPerf format reports latency percentiles + cost together. | LOW | Existing `comparison.md` includes Mean Latency and Total Cost / Cost-per-Query. Consider adding **P50/P90/P95 latency** instead of just mean (Anyscale convention) — small change, large credibility win for the latency column. |
| **Reproducibility footer**: command to regenerate, dataset version, harness version | Without this, "reproducible" is a marketing claim. ReproRAG framework requires the exact command. | LOW | Add `python -m evaluation.harness.compare` plus a pin to dataset manifest hash (`dataset/manifests/metadata.json`) and harness git SHA at top of doc. The dataset manifest pin matters because LFS contents could change. |
| **Honest "out of scope" / limitations section** | Model Cards spec (Mitchell et al. 2018) requires "Caveats & Recommendations". For our doc: state explicitly that tiers 1-3 are text-only (so multimodal scores are *expected* to be low for them, not a failure), tier-4 is the multimodal-RAG-win story, and which RAGAS metric definitions apply to which tier types. | LOW | Existing line 57 is partially there. Expand into a small "Limitations" section. |

### Differentiators (Nice-to-Have for a High-Quality Publication)

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **Multi-judge spot-check** (re-grade ~5-10 of 30 questions with a non-Gemini judge — Claude Haiku or GPT-4.1-mini — to bound the family-bias delta) | This is the *single highest-leverage credibility lift* for the blog. The Play Favorites paper (arXiv 2508.06709) literally proposes "running the same evaluation set across multiple judge models and comparing agreement rates surfaces bias profiles, with no labeled ground truth needed." Even N=5 questions × 3 tiers spot-checked with a second judge gives a directional bound — and reporting "Gemini-judge faithfulness 0.875, Claude-judge faithfulness 0.82, Δ=0.05" is a *much* stronger blog claim than "we acknowledge bias exists." | MEDIUM (cost ≈ +$0.10-0.30 for the spot-check; new code path in harness) | PROJECT.md explicitly puts mitigation out of scope. **But disclosure-via-spot-check is not mitigation, it's evidence.** Worth flagging for v1.0 negotiation: a single afternoon's work to add a `--judge` override to the existing harness and re-run a 5-question subset. If deferred, document in v1.1 backlog. |
| **Per-tier links to raw artifacts** (cost JSON, raw RAGAS scores JSON, query-by-query evaluation outputs) | LangSmith benchmarks do this with run-IDs. Lets a skeptical reader audit any single number. The artifacts already exist in `evaluation/results/{costs,metrics,queries}/` — the doc just needs to link them with relative paths. | LOW | Use relative paths (e.g., `[raw scores](./metrics/tier-3-2026-05-XX.json)`) so they resolve when readers clone the repo. The blog repo can't render these — note in the handoff that the blog should link to the GitHub permalink, not copy them. |
| **Hardware / runtime context** (laptop vs. cloud, sandbox vs. host for tier-4 MineRU, retrieval index sizes) | Anyscale LLMPerf pins this for inference benchmarks. For RAG, retrieval latency is hardware-dependent in a way generation latency is not (vector index disk access). Tier-4's MineRU-must-run-outside-sandbox constraint is a known eval-environment quirk that should be in the doc. | LOW | One-line per tier under provenance. |
| **Score range / per-question variance, not just mean** | RAGAS scores are 0-1 floats per question; reporting just the mean hides that tier-1 might have one 0.0 outlier dragging it down. A "min / mean / max" or "n_questions_with_score_below_0.5" cell catches this. | MEDIUM | Requires a small change to `compare.py` aggregator. Skip if compare.py is frozen for v1.0 — but worth a v1.1 note. |
| **Direct link to the blog post that consumes this doc** | Bidirectional citation. The blog cites `eval-numbers-v1.23.md`; the doc footer cites the blog URL + commit. Closes the reproducibility loop. | LOW | Trivial; the blog repo path is in PROJECT.md. |
| **"What changed since v1.X-1"** changelog at top | When v1.1 ships, readers (and you) need to know which numbers moved. BenchmarkCards spec encourages versioned baselines. | LOW (per-version, not v1.0) | v1.0 doesn't need this; v1.1 onwards should. |
| **Embedding-distance / cosine-similarity sanity-check column** | LangChain benchmarks publish *both* an LLM-graded score and a cosine-distance score per row. The cosine column doesn't have judge-LLM bias and acts as a tripwire — if the LLM-judge ranking and the cosine ranking disagree wildly, that's a bias signal. | MEDIUM | Adds another harness path; defer to v1.1+ unless cheap. |
| **Public dataset of the 30 golden Q&A** in the doc itself or linked | Lets readers re-run end-to-end. Already exists in repo — just link `dataset/golden_qa.json` from the doc. | LOW | Trivial. |
| **Confidence interval via bootstrap** on the mean scores | n=30 → 1000-iter bootstrap of the mean gives a 95% CI without parametric assumptions. Reports "0.875 [0.81-0.93]" instead of just "0.875". This is the *correct* response to small-N when you don't want to claim significance: report uncertainty, don't hide it. | MEDIUM (10 lines of numpy in compare.py) | High credibility win. Worth flagging — cheap and aligns with "honest magnitude reporting" framing. |

### Anti-Features (RAG-Eval Blog Posts That Ship These Lose Credibility)

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| **Cherry-picked subset reporting** ("we report on the 20 questions that worked best") | Makes numbers look better, easier blog narrative | This is **p-hacking by question selection** and is the single most common credibility-destroying move in RAG eval blog posts. Once a reader spots that the subset isn't the full golden set, all numbers are suspect. | Report all 30 questions. If a question failed for harness reasons (not tier reasons), report it as NaN with a documented cause — already the existing convention. |
| **Reporting only the headline metric** (faithfulness only, hiding context_precision) | Cleaner table, simpler blog narrative | Tier-2 has `faithfulness 0.391` but `answer_relevancy 0.799` — reporting only one metric inverts the conclusion. A reader who later finds the buried metric concludes the post was misleading. | Report all three RAGAS metrics + cost + latency in every table. If the blog body wants to focus on one, that's editorial — but the handoff doc must carry all of them. |
| **No judge model disclosure** (or buried in an appendix) | Cleaner top of post, "trust the numbers" framing | This is a **table-stakes failure**. RAGAS paper, LangChain benchmarks, Model Cards all name the judge in the header. Burying it reads as hiding something. | Judge LLM + judge embedder in the doc header, not the footer. |
| **Asymmetric judge model** (using a different judge per tier without disclosure) | Convenience — whoever ran each tier used whatever was handy | Makes tiers non-comparable. Our tier-5 already shows this risk (`openrouter/google/gemini-2.5-flash` vs the bare `google/gemini-2.5-flash` for others). Even if the underlying model is the same, the routing layer can affect determinism. | Standardize on one judge model + endpoint for the eval-date capture. If tiers were captured with different routes, **disclose the asymmetry explicitly** rather than hiding it. The current per-tier provenance row does this — preserve it. |
| **No git SHA / no model version pin** | "Latest GPT-4 / Gemini" feels current and impressive | Models silently update. A blog from 2024-Q3 citing "GPT-4" is unreproducible by 2025. ReproRAG framework calls this out as the #1 reproducibility failure. | Pin model version (`gemini-2.5-flash-001`) + git SHA + capture timestamp. Already done in existing `comparison.md` — preserve. |
| **Self-graded scores presented as objective ground truth** (e.g., "Tier 3 achieved 87.5% faithfulness") | Looks confident, blog-friendly framing | When the judge is in the same family as the generator, the number is biased upward in unknown magnitude. Asserting it as objective is dishonest. | Frame as "RAGAS-faithfulness as scored by Gemini-2.5-Flash: 0.875 (note: same model family as the tier's generator)." Verbose but honest. The existing disclaimer covers this — apply the framing in the table caption too. |
| **No NaN / failure breakdown** (just blanks or zeros) | Cleaner table | A blank cell doesn't tell the reader whether the tier scored 0 or wasn't measurable. Zeros are worse — they treat unmeasurable as "tier failed," which is unfair to tiers 4/5 in our current state. | Use `—` for "not measurable" and report the cause (`empty_contexts: 30/30`) in a NaN-breakdown subsection. Already done in `comparison.md`. |
| **Ranking tiers by a single weighted score** ("Quality score = 0.6×faithfulness + 0.3×relevancy + 0.1×precision") | Easy to put a "winner" in the blog | The weights are arbitrary and hide tradeoffs. Databricks' 60/20/20 rubric works because they justify the weights for *their* use case; for a generic blog the weights would be advocacy. | Report axes separately. Let the blog narrative argue tradeoffs in prose. The reader picks. |
| **Hidden retries / "best of N" scoring** (running each tier 3× and reporting the best run) | Tiers look better; numbers more flattering | Defensible only if disclosed (mean ± stdev across runs). Hidden, it's data fabrication. | Run once per eval-date, report once. If multi-run averaging is added in v1.1+, disclose `n_runs` per tier explicitly. |
| **Vague disclaimers** ("results may vary, this is approximate") | Looks humble | Reads as a CYA hedge that doesn't help the reader. Specific limitations are the credibility move. | The 4 specific disclaimers in existing `comparison.md` are the right pattern: name the limitation, name the affected tiers, name the source of evidence. Keep this style. |
| **Re-using the same questions for tuning AND evaluation** (overfitting the golden set during dev) | Convenient — one set of questions to think about | Out-of-scope as a v1.0 feature change, but worth disclosing if any tuning happened against the golden set. Honest blogs cite a held-out test set distinct from a dev set. | At minimum, disclose: "The 30 golden questions were used both for development inspection and final evaluation. No automated optimization (e.g., DSPy / prompt search) was run against these scores; tier prompts were authored prior to evaluation." If true. |

## Feature Dependencies

```
Frozen v1.X numbers doc
    └── requires ── Single eval-date capture (all 5 tiers re-run on same day)
                        └── requires ── Tier-4 graphml regeneration (Phase 138/139)
                        └── requires ── Tier-5 empty_contexts root cause fix
                        └── requires ── Existing comparison.md format (already produces tier rollup)

Versioned filename (v1.X)
    └── enhances ── "What changed since v1.X-1" changelog (becomes possible from v1.1)

Per-tier provenance
    └── requires ── Capture-time logging of (timestamp, git SHA, model, judge model, judge embedder)
    └── already implemented ── eval_capture writes these to costs/{tier}.json

Multi-judge spot-check (DIFFERENTIATOR)
    └── requires ── --judge override flag in evaluation harness (does not currently exist)
    └── enhances ── Self-grading bias disclaimer (turns acknowledgement into evidence)

Bootstrap CI per metric (DIFFERENTIATOR)
    └── requires ── compare.py aggregator change (~10 LOC numpy)
    └── enhances ── Sample-size disclaimer (turns "n=30 too small" into "0.875 [0.81-0.93]")

Per-question variance reporting
    └── conflicts ── "Single rolled-up score per cell" (table gets wider; pick one or use a sub-table)
```

### Dependency Notes

- **Doc requires single eval-date capture:** This is the v1.0 gating constraint already named in PROJECT.md. The doc itself is downstream of a clean multi-tier run.
- **Multi-judge spot-check enhances the bias disclaimer:** Currently the doc *acknowledges* family-bias. With a 5-question spot-check it can *bound* the bias. This is the single highest-leverage credibility lift, even though PROJECT.md marks "mitigating self-grading bias" as out of scope — *measuring* bias and *mitigating* bias are different commitments.
- **Bootstrap CI conflicts with "magnitude only" framing only superficially:** The disclaimer says "magnitude not p-values"; bootstrap CI is also a magnitude statement (an interval), not a hypothesis test. Both can coexist; the CI is more honest than a point estimate.

## MVP Definition

### Launch With (v1.0 — what the frozen doc must ship)

The minimum credible artifact:

- [ ] **Versioned filename** — `eval-numbers-v1.0.md` or `eval-numbers-v{eval-date}.md` (TBD which convention; recommend `v1.0` for semver-ish, with the eval-date in the doc header)
- [ ] **Header block**: project, eval-date, doc version, judge LLM, judge embedder, dataset manifest hash
- [ ] **Tier rollup table** — 5 rows, columns: Tier | Faithfulness | Answer Relevancy | Context Precision | Mean Latency (s) | Total Cost (USD) | Cost / Query (USD) | n | n NaN (port directly from `comparison.md`)
- [ ] **Per-question-class table** — 15 rows (5 tiers × 3 classes), columns as above (port directly)
- [ ] **Per-tier provenance block** — for each of 5 tiers: capture timestamp, generation model + version, git SHA, judge model + version, judge embedder
- [ ] **NaN breakdown** — per tier, count and cause (e.g., `tier-5: 30 empty_contexts`)
- [ ] **Honest disclaimers section** with at minimum these four: small-N, multi-hop≡cross-document for this corpus, tiers-1-3-are-text-only, judge-family-bias-acknowledged
- [ ] **Limitations section** — what the doc explicitly does NOT establish (no statistical significance, no human eval baseline, no mitigation of judge bias)
- [ ] **Reproducibility footer** — exact command to regenerate (`python -m evaluation.harness.compare` + how to re-run capture per tier), pinned dataset manifest version, repo URL, repo commit SHA
- [ ] **Link from the doc to the blog post repo path** (one-way is fine for v1.0 — the blog will add the back-link when published)

### Add After Validation (v1.1)

Triggered by blog feedback or a desire for stronger publication credibility:

- [ ] **Multi-judge spot-check** — re-grade 5 questions × 3 tiers with a non-Gemini judge (Claude Haiku or GPT-4.1-mini); report Δ in disclaimers section. *Trigger: any blog reader pushback on judge bias, or post-publication credibility lift.*
- [ ] **Bootstrap 95% CI per metric per tier** — requires ~10 LOC change in `compare.py`; reports `0.875 [0.81-0.93]` rather than `0.875`. *Trigger: cheap to add when re-touching compare.py for any reason.*
- [ ] **Latency percentiles** (P50/P90/P95) instead of just mean. Anyscale convention. *Trigger: blog reviewer asks about tail latency.*
- [ ] **Per-version changelog** at top of doc — what moved, why, since v1.X-1. *Trigger: shipping v1.1.*
- [ ] **Hardware / sandbox notes per tier** — laptop vs. cloud, retrieval index size, MineRU-runs-outside-sandbox flag for tier-4. *Trigger: any reader asks "why are these latencies so different".*

### Future Consideration (v2+)

Out of scope until product-market-fit (≡ blog publishes and gets feedback):

- [ ] **Per-question variance / outlier reporting** — min/max/stdev alongside mean. *Defer: makes table 2× wider; only justified if specific tier shows misleading mean.*
- [ ] **Embedding-distance sanity column** — LangChain-style cosine column alongside LLM-judge column. *Defer: requires new harness path; only justified if family-bias spot-check (v1.1) reveals a large delta and we want a third independent signal.*
- [ ] **Multi-run mean ± stdev** — re-run each tier N times and report run-to-run variance. *Defer: 3-5× the cost of v1.0 capture; only justified if RAGAS judge variance turns out to materially affect rankings.*
- [ ] **Human-eval baseline on a 5-question subset** — gold standard for bounding LLM-judge bias. *Defer: meaningful human time; only justified if blog gets cited by other RAG-eval work.*

**Reading the table:** Our v1.0 doc, *as already largely scaffolded by `comparison.md`*, hits more disclosure requirements than any single reference (because we report cost + latency + RAGAS metrics together, which no single benchmark above does). The two gaps relative to BenchmarkCards/Model Cards spec are:
1. **No bias-bound** — only acknowledgement (mitigated by adding the multi-judge spot-check, P2)
2. **No CI on small-N means** — only point estimates (mitigated by bootstrap CI, P2)

Both gaps are P2 features. Even without them, the v1.0 doc is at or above the disclosure floor of every published reference.

## Sources

**Academic / standards (HIGH confidence):**
- [Model Cards for Model Reporting — Mitchell et al. 2018, arXiv 1810.03993](https://arxiv.org/abs/1810.03993) — original spec; sections required for any results document
- [BenchmarkCards: Standardized Documentation for LLM Benchmarks — arXiv 2410.12974, 2024](https://arxiv.org/pdf/2410.12974) — explicitly requires self-grading-bias disclosure, sample-size limitation, judge model spec
- [RAGAS: Automated Evaluation of Retrieval Augmented Generation — arXiv 2309.15217](https://arxiv.org/html/2309.15217v1) — primary precedent for the metric set + judge-LLM-naming convention
- [Lost in the Middle — Liu et al. 2023, arXiv 2307.03172](https://arxiv.org/abs/2307.03172) — table format conventions, position-indexed reporting

**Judge-bias literature (MEDIUM-HIGH confidence — multiple recent papers agree):**
- [Play Favorites: A Statistical Method to Measure Self-Bias in LLM-as-a-Judge — arXiv 2508.06709](https://arxiv.org/abs/2508.06709) — formalizes the multi-judge spot-check pattern
